    📖 Guía de Uso — Requirement Refinator

    Todo lo que necesitas saber para sacarle el máximo provecho al sistema.

    ① Configurar el Contexto del Proyecto

    Antes de analizar HUs, el sistema necesita conocer el proyecto. Crea dos archivos en docs/contexto/:

    📄 contexto-funcional.md

    • • Nombre y descripción del proyecto
    • • Dominio de negocio y usuarios finales
    • • Reglas de negocio fundamentales
    • • Flujos de proceso principales
    • • Integraciones con sistemas externos
    • • Restricciones regulatorias / legales

    ⚙️ contexto-tecnico.md

    • • Stack tecnológico (lenguajes, frameworks)
    • • Arquitectura de microservicios
    • • Base de datos y patrones de datos
    • • Entornos: dev, qa, staging, prod
    • • APIs externas y credenciales
    • • Estándares de código del equipo

    💡 Prompt para transformar documentos existentes

    Copia este prompt en Gemini o ChatGPT junto con tus documentos del proyecto:

    Tengo los siguientes documentos de un proyecto de software [adjuntar documentos].
Necesito que extraigas y estructures la información en dos archivos Markdown:

1. contexto-funcional.md con: nombre del proyecto, dominio de negocio, usuarios finales,
   reglas de negocio, flujos principales, integraciones externas y restricciones.

2. contexto-tecnico.md con: stack tecnológico completo, arquitectura (microservicios/monolítico),
   base de datos, entornos de despliegue, APIs externas y estándares del equipo.

Usa títulos H2 para cada sección. Sé específico y conciso. Si algo no está documentado,
indica "No documentado — requiere confirmación del equipo".

    ② Analizar el Sprint con el Agente

    Una vez que tienes el contexto y las HUs en docs/HUs/Sprint-X/, ejecuta el análisis desde Claude Code / Antigravity:

    /refinar-sprint Sprint-1

    Analiza TODAS las HUs del sprint en paralelo. Genera este index.html.

    /refinar-sprint Sprint-1 --iteracion

    Re-analiza SOLO las HUs rechazadas o con feedback. Las aprobadas se preservan.

    /refinar-sprint Sprint-1 --consolidar

    Solo regenera el HTML con el data.json existente. No re-analiza.

    /generar-informe Sprint-1

    Genera el informe ejecutivo embebido en el dashboard (requiere todas las HUs aprobadas).

    ⚡ Tip de rendimiento

    El sistema lanza 1 agente por HU en paralelo. 11 HUs = ~11 invocaciones simultáneas = análisis completo en ~3-5 minutos.

    ③ Flujo de Revisión HITL (Human-in-the-Loop)

    1
    Revisar cada HU — En la tabla del Dashboard, haz clic en Revisar HU. Se abre el Focus Mode con todos los detalles.
    2
    Expandir paneles — Usa Expandir todo para ver narrativa, CAs Gherkin, tareas PERT, riesgos, dependencias y preguntas HITL.
    3
    Ajustar estimaciones PERT — Edita O/P/Pe en la tabla de tareas. E = (O + 4P + Pe) / 6 se recalcula automáticamente.
    4
    Responder preguntas HITL — El agente generó preguntas de clarificación. Respóndelas con el PO antes de iniciar desarrollo.
    5
    Aprobar o Rechazar — Si la HU está lista, Aprobar. Si necesita mejoras, Rechazar y escribe el feedback. Al aprobar todas, se ofrece descargar el resumen refinado.
    6
    Exportar respaldo — Usa Respaldo → Exportar Markdown. El estado vive en localStorage — si limpias el navegador lo perderás. Exporta con frecuencia.
    7
    Iterar con feedback — Ejecuta /refinar-sprint Sprint-X --iteracion para re-analizar solo las HUs rechazadas.

    ⚠ Estado en localStorage — riesgo de pérdida

    El estado de feedback, PERT y mediciones EVM vive en el localStorage del navegador. Si limpias la caché o abres el archivo en otro navegador, el estado se pierde. Exporta el respaldo Markdown periódicamente (mínimo al final de cada sesión de revisión).

    ④ Monitoreo EVM — Avance del Sprint

    Una vez aprobadas las HUs, ve a Avance del Sprint → EVM para monitorear el progreso.

    ① Declara las fechas

    Ingresa inicio y fin del sprint. Usa ⚡ Auto para calcular el fin automáticamente excluyendo festivos Colombia.

    ② Ajusta el BAC

    El BAC viene del PERT. Si hay ajustes de alcance, edita el campo BAC editable en horas.

    ③ Registra mediciones

    Al cierre de cada día (mínimo 2×/semana), ingresa PV, EV y AC por HU. Usa 📌 Guardar medición para crear un snapshot.

    ④ Genera Bitácora PMO

    El botón 📋 Generar Bitácora PMO (PDF) abre un reporte en modo claro listo para imprimir o enviar al cliente.

    📊 Interpretar indicadores EVM

    CPI > 1.0 = bajo presupuesto · SPI > 1.0 = adelantado · TCPI > 1.2 = muy difícil de recuperar · EAC = estimación de costo final del sprint

    ⑤ Cierre del Sprint

    • ✅
      Medición final EVM — Registra la medición del último día con EV=BAC si completaste el sprint.
    • ✅
      Revisar riesgos — En Radar de Riesgos, marca como cerrados los riesgos que no se materializaron.
    • ✅
      Exportar respaldo final — Respaldo → Exportar Markdown. Guarda este archivo en Google Drive o el repositorio.
    • ✅
      Bitácora PMO final — Genera el reporte PDF y comparte con el cliente / PMO.
    • ✅
      Próximo sprint — Crea la carpeta docs/HUs/Sprint-2/, agrega las nuevas HUs y ejecuta /refinar-sprint Sprint-2.
